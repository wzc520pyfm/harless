# harless v0.1 — Design Spec

- **Date:** 2026-04-20
- **Status:** Approved (brainstorming complete)
- **Author:** harless project
- **Audience:** future implementers, contributors, and adopters

---

## 1. Project Statement

`harless` is an **agent-agnostic, minimal AI Coding Harness kit**. Through `npx harless init` / `add`, it injects 8 capability modules into a user's project as plain markdown files and shell scripts. Any AI agent that respects the `[AGENTS.md](https://agents.md)` convention (Claude Code, Cursor, Codex CLI, Copilot CLI, Gemini CLI, …) gains an immediate, complete production loop:

> Skills · Spec · Loop · Memory · Browser-Debug · Orchestration · Review · Simplify

The goal is not to replace heavy alternatives (Superpowers, OpenSpec, claude-mem, agentation, etc.) but to provide the **smallest possible complete subset** that global Web developers can adopt at near-zero cost, and that is easy to learn, extend, and fork.

### 1.1 Non-goals (v0.1)

- Replacing Superpowers / OpenSpec / claude-mem feature-by-feature
- Self-built MCP server (deferred to v0.2+)
- Slash commands, hooks, plugins, subagent-specific configurations
- SaaS backend, telemetry, accounts, online dashboards
- Multi-OS matrix beyond macOS + Linux (Windows deferred to v0.2)
- A spec validator (deferred to v0.2+)

## 2. Locked Architectural Decisions


| #   | Decision                                                                      | Rationale                                                          |
| --- | ----------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| D1  | **Agent-agnostic** via `AGENTS.md` + Read tool + non-interactive CLI          | Avoids per-agent adapter sprawl; serves global developers          |
| D2  | **CLI-injected files (shadcn/OpenSpec style)** via `npx harless init` / `add` | Code lives in user's repo; trivial fork & customize                |
| D3  | **Two primitives only**: Markdown + Shell/Node scripts                        | Keeps the surface tiny and learnable                               |
| D4  | External MCP only — reuse `chrome-devtools-mcp`, no self-built MCP            | YAGNI; community already maintains it                              |
| D5  | **Web-first defaults + generic degradation**                                  | Auto-detect Next/Vite/React; degrade via `--stack=generic`         |
| D6  | **Smoke-tested on Tier 1 = CC, Cursor, Codex CLI**                            | Three formats represent the field; others are AGENTS.md-compatible |
| D7  | **No daemons, no hooks, no global config writes**                             | Zero side effects on user's machine                                |
| D8  | Project name = `**harless`**                                                  | Short, unique, self-evidently "harness minus weight"               |


### 2.1 Five Design Axioms (Inviolable)

1. **Zero runtime** — no daemons, no port listeners, no background workers.
2. **Two primitives** — only Markdown files and shell/Node scripts; external MCP only via reuse.
3. **Code belongs to the user** — files written by `add` are user-owned; `update` uses a 3-way diff merge, never a clobber.
4. **AGENTS.md is the protocol** — all cross-agent compatibility flows through one file at the project root.
5. **YAGNI ruthlessly** — no slash commands, no self-MCP, no plugins, no GUI, no SaaS in v0.1.

## 3. Three-Layer Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  Agent Layer (the user's)                                    │
│  Claude Code  │  Cursor  │  Codex CLI  │ (other AGENTS.md)   │
└──────────────────────────┬──────────────────────────────────┘
                           │ reads
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Convention Layer (injected into user repo)                  │
│  ./AGENTS.md            ── single entry point                │
│  ./.mcp.json + ./.cursor/mcp.json  ── MCP (Claude + Cursor)  │
│  ./.agents/                                                 │
│    config.json                                               │
│    skills/{brainstorming,tdd,…}/SKILL.md   (5 disciplines)   │
│    {spec,loop,memory,browser-debug,                          │
│     orchestrate,review,simplify}/SKILL.md  (7 modules)       │
│    specs/<change>/{spec,plan,tasks}.md                       │
│    loop/<task>/{goal.md,check.sh,iter-N.log}                 │
│    memory/{sessions,topics}/*.md                             │
│    {orchestrate,review,simplify}/<ts>.{log,md}               │
│    scripts/{loop,compact,review,dispatch,simplify}.sh        │
└──────────────────────────┬──────────────────────────────────┘
                           │ generated by
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Tool Layer (npm package)                                    │
│  harless CLI — ≤ 500 LOC Node                                │
│   init / add / remove / update / doctor                      │
└─────────────────────────────────────────────────────────────┘
```

**Property:** if the user uninstalls the `harless` CLI, their project keeps working. The Convention Layer is the asset; the Tool Layer is just a copier.

## 4. CLI Surface

### 4.1 Commands


| Command                   | Purpose                                                 | Idempotent    |
| ------------------------- | ------------------------------------------------------- | ------------- |
| `harless init`            | Detect stack, interactively select modules, write files | ✅             |
| `harless add <module>`    | Add one module after init                               | ✅             |
| `harless remove <module>` | Remove one module (with hash-aware safety)              | ✅             |
| `harless update`          | Hash-based update with `[k]eep / [o]verwrite` per file  | ✅             |
| `harless doctor`          | Inspect environment, agent config, MCP, permissions     | ✅ (read-only) |


**Global flags (apply to all commands):**

- `--yes` — non-interactive; semantics per §4.3.
- `--offline` — skip the optional npm-registry version check and any other network probe (e.g., MCP package metadata lookup). Useful in air-gapped CI. Has no effect on file operations.

**Not in v0.1:** `login`, `publish`, `share`, `ai`, `run`, `watch`, `spec new`, `memory query`. The CLI is **not** a second agent UI — agents call scripts and read files directly.

### 4.2 Exit-Code Convention (consolidated)


| Exit  | Meaning                                                                   | Used by                           |
| ----- | ------------------------------------------------------------------------- | --------------------------------- |
| `0`   | success                                                                   | all commands                      |
| `1`   | unrecoverable error (filesystem, network, malformed config)               | all commands                      |
| `2`   | unresolved conflict / user input required but `--yes` was set             | `init`, `add`, `update`, `remove` |
| `3`   | precondition failed (Node version, missing dependency)                    | all commands                      |
| `10`+ | `doctor` only: bitmask of failing checks (10=AGENTS, 11=MCP, 12=perms, …) | `doctor`                          |


### 4.3 `--yes` Semantics per Command


| Command      | With `--yes`                                                                          | Without                                              |
| ------------ | ------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| `init`       | use defaults; on AGENTS.md/MCP merge conflict → exit 2                                | interactive prompts                                  |
| `add <m>`    | overwrite-if-pristine, exit 2 if user-modified                                        | prompt `[k]eep / [o]verwrite`                        |
| `update`     | overwrite-if-pristine, exit 2 if conflict                                             | prompt `[k]eep / [o]verwrite`                        |
| `remove <m>` | delete pristine files; move user-modified files to `.agents/.trash/<ts>/` (per §4.6) | per-file `[d]elete / [t]rash / [k]eep` prompt (§4.6) |
| `doctor`     | n/a (read-only)                                                                       | n/a                                                  |


### 4.4 `init` Behavior

1. Detect stack (`package.json` heuristics: `next`, `vite`, `react`, `remix`, `astro`, `svelte`).
2. Detect package manager (`pnpm-lock.yaml` / `yarn.lock` / `package-lock.json`).
3. **Monorepo handling**: if a workspaces field is detected (`pnpm-workspace.yaml`, `package.json#workspaces`, `lerna.json`), `init` runs at the **workspace root**; exits 3 (PRECONDITION) if invoked from inside a sub-package, with override flag `--here`.
4. Interactive multi-select: 6 modules pre-checked (`skills, spec, loop, memory, browser-debug, review`), 2 unchecked (`orchestrate, simplify`).
5. Interactive multi-select: smoke-tested agent (CC, Cursor, Codex CLI, Other).
6. Write files; merge `AGENTS.md` and `.mcp.json` if present.
7. Print "Next steps" with verification command and a `.gitignore` snippet suggestion (see §13.3).

**Non-interactive form:** `--yes --modules=... --agents=...` for CI.

### 4.5 `update` — Hash-Based Per-File Action (v0.1: keep / overwrite only)

Each templated file's SHA-256 is recorded in `.agents/config.json` at install time. On `update`:


| Disk hash vs old template | New template vs old template | v0.1 action                                           |
| ------------------------- | ---------------------------- | ----------------------------------------------------- |
| same                      | changed                      | silent overwrite                                      |
| changed                   | same                         | leave disk untouched                                  |
| changed                   | changed                      | print unified diff; user picks `[k]eep / [o]verwrite` |
| missing                   | n/a                          | offer to recreate                                     |


`**[m]erge` (3-way text merge via `git merge-file`) is deferred to v0.2.** v0.1 deliberately ships only the two safe options to keep the CLI ≤ 500 LOC and the conflict UX trivially understandable. Users who want a real merge can manually run `git merge-file` themselves; the diff output prints both sides verbatim.

**No automatic clobber on conflict, ever.**

### 4.6 `remove` Behavior

For each file the module owns:

1. If file's hash matches recorded template hash → delete (file was never edited).
2. If file's hash differs → file is user-modified:
  - Without `--yes`: prompt `[d]elete / [t]rash / [k]eep` (default trash).
  - With `--yes`: move to `.agents/.trash/<YYYY-MM-DD-HH-MM-SS>/` (preserving relative path), then update `config.json`.
3. Always remove the corresponding entries from `AGENTS.md`'s Skills Index table and the module's section in `config.json`.

Trash is never auto-purged in v0.1; user can `rm -rf` when ready.

### 4.7 `doctor` Checks (v0.1: snippet-printer, not runtime probe)

- Node ≥ 18 (probed)
- `AGENTS.md` exists at root and contains the `harless` managed block (probed)
- `.agents/scripts/*.sh` are executable (probed)
- If `browser-debug` enabled: `.mcp.json` lists `chrome-devtools-mcp` (probed); **Chrome reachability is NOT probed** in v0.1 — too platform-specific. Doctor only prints the install command.
- For each declared agent: print the **paste-ready snippet** for that agent's MCP config path (Codex `~/.codex/config.toml`, Gemini `.gemini/settings.json`, Copilot config). **Doctor does not write or read those files** — privacy and YAGNI.

Each check returns ✓ / ⚠ / ✗ with a single-line remediation. Bitmask in exit code (see §4.2).

## 5. File Layout

### 5.1 Injected into User Repo

The 8 capability **modules** are: `skills, spec, loop, memory, browser-debug, orchestrate, review, simplify`. Each module owns a subdirectory under `.agents/`. Seven modules contain a single `SKILL.md`; the `skills` module is special and contains 5 sub-discipline SKILLs.

```
<user-project>/
├─ AGENTS.md                           # single entry; merged
├─ .mcp.json                           # merged
└─ .agents/
   ├─ config.json                      # CLI metadata
   ├─ README.md                        # one-page handover
   │
   ├─ skills/                          # module: skills (5 disciplines bundle)
   │  ├─ brainstorming/SKILL.md
   │  ├─ tdd/SKILL.md
   │  ├─ systematic-debugging/SKILL.md
   │  ├─ verification-before-completion/SKILL.md
   │  └─ receiving-review/SKILL.md
   ├─ spec/SKILL.md                    # module: spec
   ├─ loop/SKILL.md                    # module: loop
   ├─ memory/SKILL.md                  # module: memory
   ├─ browser-debug/SKILL.md           # module: browser-debug
   ├─ orchestrate/SKILL.md             # module: orchestrate
   ├─ review/SKILL.md                  # module: review
   ├─ simplify/SKILL.md                # module: simplify
   │
   ├─ specs/                           # spec module's working area
   │  ├─ .template/{spec,plan,tasks}.md
   │  └─ <YYYY-MM-DD-topic>/{spec,plan,tasks}.md
   │
   ├─ loop/                            # loop module's working area (shared dir)
   │  ├─ .template/{goal.md,check.sh}  # 5-line skeletons
   │  └─ <task>/{goal.md,check.sh,iter-N.log}
   │
   ├─ memory/                          # memory module's persistence
   │  ├─ sessions/<YYYY-MM-DD>.md      # append-only daily log
   │  └─ topics/<topic>.md             # curated topic files
   │
   ├─ orchestrate/                     # parallel run logs
   │  └─ <task-id>-<ts>.log
   │
   ├─ review/                          # review output history
   │  └─ <ts>.md
   │
   └─ scripts/                         # all executable capability
      ├─ loop.sh
      ├─ compact.sh
      ├─ review.sh
      ├─ dispatch.sh
      └─ simplify.sh                   # default-disabled module
```

Note `loop/` and `loop/SKILL.md` coexist: the SKILL.md is the module entry point; sibling subdirectories under `loop/` are runtime working dirs. This is unambiguous because `SKILL.md` is a fixed filename. (Same applies to `memory/SKILL.md`, `orchestrate/SKILL.md`, `review/SKILL.md`.)

### 5.2 Size Budget


| Type                                    | Files | Max LOC each | Total budget   |
| --------------------------------------- | ----- | ------------ | -------------- |
| Discipline `SKILL.md` (under `skills/`) | 5     | 100          | ≤ 500          |
| Module `SKILL.md` (the other 7)         | 7     | 100          | ≤ 700          |
| `scripts/*.sh`                          | 5     | 80           | ≤ 400          |
| `specs/.template/*`                     | 3     | 50           | ≤ 150          |
| `loop/.template/*`                      | 2     | 10           | ≤ 20           |
| `AGENTS.md` harless block               | 1     | 60           | ≤ 60           |
| **Injected total**                      |       |              | **≤ 1830 LOC** |
| CLI source                              | —     | —            | **≤ 500 LOC**  |
| **Whole project**                       |       |              | **≤ 2330 LOC** |


This is the concrete commitment behind "minimal". Note: budget tightened from 120 LOC/SKILL to 100 LOC/SKILL to absorb the additional discipline files; total stays in the same order of magnitude.

### 5.3 `AGENTS.md` Injected Block

Marker-delimited so updates and user edits coexist:

```markdown

Any AGENTS.md-compliant agent should read the relevant skill file on

### Disciplines (always-on principles, read whenever in doubt)

1. Before non-trivial work, check the Skills Index and read the matching SKILL.md.
```

### 5.4 SKILL.md Frontmatter

```yaml
---
name: loop
description: Use when you need an iterative self-correcting loop that keeps running an agent against a goal until a done condition is met. Involves writing a goal.md and check.sh, then calling .agents/scripts/loop.sh.
when-to-use:
  - Long-running refactors with a clear done criterion
  - Fixing flaky tests by repeated targeted runs
when-not-to-use:
  - Creative/design work (use brainstorming skill instead)
  - Single-shot tasks

  - .agents/scripts/loop.sh
---
```

`name` + `description` **extend** Anthropic's Skills spec format (`when-to-use` / `when-not-to-use` / `requires-scripts` are harless-specific additions). The base fields are intended to work with CC's and Cursor's native Skill tools where supported (see §7.5 verification gate); agents without native skill systems benefit through the AGENTS.md index above.

## 6. Eight Modules

Each module is **1 SKILL.md + (optional) 1 script**, ≤ 100 LOC markdown and ≤ 80 LOC script (per §5.2 budget table). The `skills` module is a 5-file bundle; all other modules ship a single SKILL.md.

### 6.1 `skills` — discipline bundle (Superpowers minimal subset)

This module is a **bundle of 5 sub-discipline SKILL files** rather than a single SKILL.md. Selected for "agent wouldn't think of it + biggest quality impact":

- `.agents/skills/brainstorming/SKILL.md` (intent before code)
- `.agents/skills/tdd/SKILL.md` (test-first discipline)
- `.agents/skills/systematic-debugging/SKILL.md` (no random guesses)
- `.agents/skills/verification-before-completion/SKILL.md` (evidence over assertions)
- `.agents/skills/receiving-review/SKILL.md` (structured response to review feedback)

Each ≤ 100 LOC. AGENTS.md exposes them under "Disciplines" (always-on) rather than "Capability Modules" (on-demand) — see §5.3. No script.

### 6.2 `spec` — Spec-Driven Development (OpenSpec minimal)

Three-step workflow: idea → `spec.md` (what) → `plan.md` (how) → `tasks.md` (atomic tasks). Templates only; no validator in v0.1. Agent copies `.template/` to a dated change folder. Single `SKILL.md` at `.agents/spec/SKILL.md`.

### 6.3 `loop` — Ralph self-correcting loop

`.agents/loop/.template/` ships with a 5-line `goal.md` skeleton and a 3-line `check.sh` (`#!/usr/bin/env bash` + a stub). `loop/SKILL.md` instructs the agent to `cp -r .template/ <task>/`, fill in goal+check, then run the script.

`scripts/loop.sh` (≤ 40 LOC bash):

```bash
#!/usr/bin/env bash
# AGENT_CMD examples:
#   AGENT_CMD="claude -p"          (default)
#   AGENT_CMD="cursor-agent -p"
#   AGENT_CMD="codex exec"
set -euo pipefail
WORK="${1:?usage: loop.sh <work-dir>}"; cd "$WORK"
MAX="${MAX_ITER:-20}"
AGENT_CMD="${AGENT_CMD:-claude -p}"
[[ -f goal.md && -f check.sh ]] || { echo "need goal.md + check.sh in $WORK"; exit 3; }
for i in $(seq 1 "$MAX"); do
  echo "=== iter $i ==="
  $AGENT_CMD "$(cat goal.md)" | tee "iter-$i.log"
  if bash check.sh; then echo "✓ done @ iter $i"; exit 0; fi
done
echo "✗ max iterations reached"; exit 1
```

### 6.4 `memory` — cross-session memory (claude-mem minimal, zero daemon)

**I/O contract** (resolves the "fresh process can't read transcript" issue):

- The **calling agent** authors the summary in markdown using its in-context conversation. The script does NOT call any LLM.
- `**scripts/compact.sh`** is a thin appender: it reads markdown from **stdin**, prepends a timestamp header, and appends to `.agents/memory/sessions/<YYYY-MM-DD>.md`.
- Topic files in `.agents/memory/topics/<topic>.md` are written by the agent **directly** (using its Write tool); no script involved. The agent picks topic names matching keywords used at session start.
- **Read**: SKILL.md instructs the agent to `ls .agents/memory/topics/` and `grep` for relevant terms at session start.
- **Filename = retrieval key** (`auth.md`, `deploy-netlify.md`, …).
- **Secret protection**: SKILL.md's redaction checklist runs before any write; README recommends gitignoring `memory/sessions/`.
- **Log rotation** (footgun mitigation): SKILL.md instructs the agent to summarize-and-truncate any `sessions/<date>.md` file exceeding 500 lines into the corresponding topic files, then leave a one-line stub. No automatic rotation in v0.1.

`scripts/compact.sh` (≤ 25 LOC bash):

```bash
#!/usr/bin/env bash
# Usage:
#   echo "<markdown synopsis>" | bash .agents/scripts/compact.sh
#   bash .agents/scripts/compact.sh < summary.md
set -euo pipefail
DATE=$(date +%Y-%m-%d); TS=$(date +%Y-%m-%dT%H:%M:%S%z)
DIR=.agents/memory/sessions; mkdir -p "$DIR"
FILE="$DIR/$DATE.md"
{ echo; echo "## Session @ $TS"; echo; cat; echo; } >> "$FILE"
echo "✓ appended to $FILE"
echo "Reminder: update topic files in .agents/memory/topics/ for any reusable knowledge."
```

### 6.5 `browser-debug` — Web debugging via `chrome-devtools-mcp`

- Zero self-built code; we just merge `chrome-devtools-mcp` into `.mcp.json`.
- The skill's value is **discipline**: snapshot before action, scroll-into-view before click, abandon after 4 failed attempts.

### 6.6 `orchestrate` — parallel subagent dispatch (agentation minimal)

**Input format**: `tasks.json` — array of objects `{ "id": "string", "prompt": "string", "cwd": "string?", "timeoutSec": number? }`. JSON chosen over YAML to avoid an extra dep (per §11 decision).

**Output**: per-task log at `.agents/orchestrate/<id>-<ts>.log` (stdout+stderr combined). The script **never interleaves output to terminal** — it prints a one-line "started" per task, then a final summary table with paths and exit codes. This solves the "N parallel agents shouting into one terminal" problem.

`scripts/dispatch.sh` (≤ 60 LOC bash, parses JSON inline with 5 lines of node):

```bash
#!/usr/bin/env bash
set -euo pipefail
TASKS="${1:?usage: dispatch.sh <tasks.json>}"
AGENT_CMD="${AGENT_CMD:-claude -p}"
TS=$(date +%Y-%m-%dT%H-%M-%S)
LOG_DIR=.agents/orchestrate; mkdir -p "$LOG_DIR"
node -e '
  const t = JSON.parse(require("fs").readFileSync(process.argv[1], "utf8"));
  for (const x of t) console.log([x.id, x.prompt, x.cwd||".", x.timeoutSec||0].join("\t"));
' "$TASKS" | while IFS=$'\t' read -r ID PROMPT CWD TIMEOUT; do
  LOG="$LOG_DIR/$ID-$TS.log"
  echo "→ $ID (cwd=$CWD, log=$LOG)"
  ( cd "$CWD" && timeout "${TIMEOUT}s" $AGENT_CMD "$PROMPT" ) >"$LOG" 2>&1 &
done
wait
echo
# Final summary table schema (always emitted):
#   id <TAB> exit <TAB> log
# Sorted by exit code descending (failures first); exit code of dispatch.sh
# itself is 0 if all tasks exit 0, else 1.
printf "%-20s %-6s %s\n" "id" "exit" "log"
```

v0.1: parallel-only, no DAG. DAGs decompose into multiple sequential dispatches. Default-off in `init`.

### 6.7 `review` — independent diff review

`scripts/review.sh` (≤ 60 LOC bash) spawns a fresh `$AGENT_CMD` process with **only the checklist + diff** (no main-session context bias). Output goes to `.agents/review/<ts>.md` (per §6.9 rule #3, **not** `.agents/review-<ts>.md`):

```bash
#!/usr/bin/env bash
set -euo pipefail
REF="${1:-HEAD}"
AGENT_CMD="${AGENT_CMD:-claude -p}"
TS=$(date +%Y-%m-%dT%H-%M-%S)
OUT_DIR=.agents/review; mkdir -p "$OUT_DIR"
OUT="$OUT_DIR/$TS.md"
DIFF=$(git diff "$REF")
CHECKLIST=$(cat .agents/review/SKILL.md)
{ echo "$CHECKLIST"; echo; echo "=== DIFF ==="; echo "$DIFF"; } | $AGENT_CMD > "$OUT"
echo "✓ wrote $OUT"
```

Independent process (not subagent) keeps it cross-agent.

### 6.8 `simplify` — code simplification scan

Pure-skill primarily; `scripts/simplify.sh` is an accelerator following the same independent-agent pattern as `review`. Output goes to `.agents/simplify/<ts>.md`. Default-off in `init` but **the script is always shipped** if the module is added (kept ≤ 30 LOC).

### 6.9 Common Script Conventions

All scripts honor:

1. `$AGENT_CMD` env (default `claude -p`) for cross-agent. Each script's header lists 3 example invocations (CC / Cursor / Codex).
2. `set -euo pipefail` for fail-loud behavior.
3. Output artifacts to `.agents/<module>/<timestamp>.{log,md}` (never to `.agents/<module>-<timestamp>` at the root).
4. Exit codes per §4.2 (0 success, 1 unrecoverable, 2 conflict-with-`--yes`, 3 precondition).
5. No deps beyond `git`, `bash`, `node`. No `jq`/`yq`/`parallel` — small inline alternatives.

## 7. Cross-Agent Compatibility

### 7.1 Capability Matrix


| Agent       | Reads AGENTS.md | Read tool | Terminal tool | Non-interactive CLI | MCP path                          |

| Claude Code | ✅               | ✅         | ✅             | `claude -p`         | `.mcp.json`                       |

| Codex CLI   | ✅               | ✅         | ✅             | `codex exec`        | `~/.codex/config.toml` (global)   |

| Gemini CLI  | ✅               | ✅         | ✅             | `gemini -p`         | `.gemini/settings.json`           |


### 7.2 Four Differences We Resolve

**Difference 1 — MCP config paths.** Write **`.mcp.json`** (Claude Code) and **`.cursor/mcp.json`** (Cursor) with the same merged `mcpServers` object; harless only adds missing keys. For Codex/Copilot/Gemini, print a paste-snippet in `.agents/README.md`. **Never modify global config files.**

**Difference 2 — `$AGENT_CMD` syntax.** Stored in `config.json:defaultAgentCmd`; scripts honor env override. Default = `claude -p`. Each script's header lists alternative invocations.

**Difference 3 — Skill discovery vs manual reference.** SKILL.md frontmatter benefits CC/Cursor's native discovery. AGENTS.md's "Skills Index" table benefits agents without native skill systems. Same SKILL.md serves both.

**Difference 4 — AGENTS.md coexistence with other tools.** Marker-delimited managed block (`<!-- BEGIN/END harless v0.1 -->`). `update` only touches that block. If openspec/superpowers blocks are detected, `doctor` notes coexistence and links to a reconciliation doc.

### 7.3 Verification Gate (release red line)

Run on each Tier 1 agent: ask `"Please help me turn this idea into a design spec: a todo list app."` Expected: agent reads `.agents/skills/brainstorming/SKILL.md` **before** acting, then follows it. **A failure on any Tier 1 agent blocks release.**

### 7.4 Compatibility Tiers

- **Tier 1 (smoke-tested):** Claude Code, Cursor, Codex CLI
- **Tier 2 (AGENTS.md-compatible, community-verified):** Copilot CLI, Gemini CLI
- **Tier 3 (best-effort):** any agent reading AGENTS.md with shell + Read tools

## 8. Lifecycle & Data Flow

### 8.1 Canonical Developer Day


| Time    | Event                     | Agent action                                                                                                                                   |
| ------- | ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| T+00:00 | `npx harless init`        | —                                                                                                                                              |
| T+00:05 | "Add a favorites feature" | reads `.agents/skills/brainstorming/SKILL.md` → produces `.agents/specs/2026-04-20-favorites/{spec,plan,tasks}.md`                           |
| T+00:30 | implementation            | reads `.agents/spec/SKILL.md` → tasks → TodoWrite → reads `.agents/skills/tdd/SKILL.md`; reads `.agents/browser-debug/SKILL.md` for UI bugs |
| T+01:00 | flaky tests               | reads `.agents/loop/SKILL.md` → writes `goal.md`+`check.sh` → `loop.sh`                                                                       |
| T+01:30 | pre-commit                | reads `.agents/review/SKILL.md` → `review.sh HEAD` → reads `.agents/review/<ts>.md`                                                          |
| T+01:45 | session end               | reads `.agents/memory/SKILL.md` → `compact.sh` → updates `.agents/memory/topics/favorites.md`                                                |
| T+Day+1 | next session              | reads AGENTS.md → memory → continues without re-asking                                                                                         |


Six of eight modules engaged in a single canonical day. `orchestrate` and `simplify` arrive when needed via `harless add`.

### 8.2 Artifact Dependency Graph

```
              AGENTS.md (root)
                   │
    ┌──────────────┼─────────────────┐
    ▼              ▼                 ▼
 skills/*    specs/<change>/    scripts/*
                   │                 │
                   ▼                 ▼
              loop.sh ──► loop/<task>/iter-N.log
              review.sh ──► review/<ts>.md
                   │            │
                   ▼            ▼
              memory/topics/*.md ◄── compact.sh
```

All arrows point to plain files. No process is in the loop.

### 8.3 Session Phases


| Phase | Trigger                                            | Required action                                                   | Artifacts                                                           |
| ----- | -------------------------------------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------- |
| Start | first user message                                 | read AGENTS.md; if `memory` enabled, `ls memory/topics/` and grep | `memory/topics/` (read)                                             |
| Mid   | each task                                          | route via Skills Index; call scripts via `$AGENT_CMD`             | `specs/`, `loop/`, `review/*.md`                                    |
| End   | user says "done" / completes a TodoWrite root task | `compact.sh`                                                      | `memory/sessions/<date>.md` (append), `memory/topics/*.md` (upsert) |


Session end is **not auto-detected**. SKILL.md tells the agent which signal words and structural completions trigger compaction. This deliberately avoids hooks (which would break agent-agnosticism).

### 8.4 Failure Modes (decision matrix)


| Scenario                            | v0.1 behavior                                                              | Not done (over-engineering)             |
| ----------------------------------- | -------------------------------------------------------------------------- | --------------------------------------- |
| `loop.sh` exhausts MAX_ITER         | exit 1, prints last log path                                               | no auto prompt-rewrite, no agent swap   |
| `review.sh` finds severe issue      | writes `.agents/review/<ts>.md`; main agent decides                       | no commit blocking, no git hook         |
| `compact.sh` may include secrets    | redaction checklist + README gitignore tip                                 | no automatic secret scanner (v0.2+)     |
| Co-installed with OpenSpec          | `.specs/` and `.openspec/` coexist independently                           | no migration tool                       |
| `chrome-devtools-mcp` version drift | `doctor` prints versions and upgrade hint                                  | no version pinning, no vendoring        |
| User edited a SKILL file            | `update` prints unified diff; user picks `[k]eep / [o]verwrite` (per §4.5) | no "protected mode", no intent tracking |
| Tier 1 agent inconsistency          | release blocked; Tier 2/3 → issue                                          | no runtime capability probing           |


Stance: **observable + user-decided** beats **automatic fallback**.

## 9. Error Handling, Testing, Distribution

### 9.1 Error-Handling Principles

1. **Fail loud, fail early** — `process.exitCode` in CLI; `set -euo pipefail` in scripts; preconditions in SKILLs.
2. **Never silently mutate** — show a path before any write; require explicit choice on conflict; `--yes` exits 2 on conflict.
3. **Recoverable by hand** — delete `.agents/` and re-`init` always recovers; everything is plain text in git.

### 9.2 Test Pyramid

**Unit (vitest):** `mergeAgentsMd`, `mergeMcpJson`, `decideUpdateAction` (the v0.1 hash-compare helper that returns `keep | overwrite | conflict | recreate`), `detectStack`. Coverage ≥ 80% line, ≥ 70% branch. AGENTS.md rendering uses snapshot tests.

**Integration (`execa` + temp dirs):** 8 mandatory scenarios:

1. Empty dir `init` (default selection)
2. `init` over existing AGENTS.md containing an `openspec` block
3. `init → add orchestrate → remove orchestrate`
4. `init` → user edits one SKILL → `update` (template unchanged) → user edit preserved
5. `init` → user edits one SKILL → `update` (template also changed) → conflict path; `--yes` exits 2
6. Re-`init` (idempotency)
7. `doctor` exit codes across 7 scenarios (all green / missing perms / missing MCP / missing AGENTS block / etc.)
8. `.mcp.json` with other servers + add `chrome-devtools-mcp` (merge correctness)

**E2E (manual, recorded):** per Tier 1 agent, a checklist + screen recording in `docs/smoke/<agent>.md`. All three must be green to release.

**Out of scope:** real agent CI, semantic LLM-output assertions, multi-OS matrix.

### 9.3 Repository Layout

```

├─ package.json (workspaces)
├─ packages/cli/
│  ├─ src/
│  │  ├─ bin.ts
│  │  ├─ commands/{init,add,update,remove,doctor}.ts
│  │  ├─ lib/{merge,detect,hash,prompts,logger}.ts
│  │  └─ templates/

│  │     └─ modules/
│  │        ├─ skills/…

│  │        └─ …
```

**Constraints:** single npm package (`harless`), templates are the product (1:1 of what users get), CLI ≤ 500 LOC.

### 9.4 Versioning, Release, Upgrade

- **SemVer.** v0.x may break template structure; v1.0+ guarantees ≥ 2-minor migration paths.
- **Release:** tag `v0.x.y` → CI runs unit + integration → publish npm. E2E recordings must be merged into `docs/smoke/` first (manual gate).
- **Upgrade nudge:** silent npm-registry version check on each invocation (< 1 s timeout, fails silent). One-line notice if behind. **No telemetry.**
- **Deprecation:** any template-file removal must warn through `update` for at least one minor before removal.

### 9.5 Documentation

VitePress site + README:

- README — 10-minute onboarding (install → init → first brainstorm → first loop)
- Getting Started — same, longer, with troubleshooting
- One page per module
- "Compared with other Harnesses" — honest positioning vs Superpowers / OpenSpec / claude-mem / agentation
- "Extending harless" — how to remove / add / fork modules

**Out of scope:** interactive playground, online config builder, blog, SaaS console.

### 9.6 Privacy & Compliance

- **Zero telemetry.** No data leaves the user's machine. The CLI itself makes one optional version check against `registry.npmjs.org` (1 s timeout, fails silent, suppress with `--offline`).
- README states this prominently: "no telemetry, no account; the CLI's only network call is an optional npm-registry version check that you can disable with `--offline`."

### 9.7 License & Governance

- **MIT** — most permissive, aligned with "fork freely".
- **Governance** — single maintainer in v0.1; CONTRIBUTING covers issue/PR rules; no SLA during v0.x.
- **Code of Conduct** — Contributor Covenant (one file).

## 13. Operating Edge Cases (v0.1 covered behavior)

### 13.1 Monorepo

- `init` is workspace-aware: detects `pnpm-workspace.yaml`, `package.json#workspaces`, `lerna.json`. Runs at the workspace root and exits 3 (PRECONDITION) if invoked from a sub-package (override: `--here`).
- `.agents/` lives **only at the workspace root**; per-package `.agents/` directories are not supported in v0.1 (and are flagged as a warning by `doctor`).

### 13.2 Coexistence with `CLAUDE.md` / `.cursorrules` / Other AI Convention Files

- `init` detects existing `CLAUDE.md`, `.cursorrules`, `.github/copilot-instructions.md`. **Does not modify them.**
- It writes `AGENTS.md` (creating or merging) and prints a one-paragraph note in the "Next steps" output: "AGENTS.md is the cross-agent standard; you may want to add a one-line `@AGENTS.md` reference to your existing convention files."
- `doctor` lists detected legacy files and reminds the user.

### 13.3 `.gitignore` Suggestions

`init` prints (not writes) a suggested `.gitignore` block:

```
# harless: optional, depending on your team's policy
.agents/.trash/
.agents/memory/sessions/   # uncomment if sessions may contain secrets
.agents/orchestrate/
.agents/review/
.agents/simplify/
```

It does **not** modify `.gitignore` automatically — that file is too sensitive to mutate unprompted.

### 13.4 Offline / Air-Gapped Behavior

- `npx harless@latest` requires network on first invocation; subsequent invocations use the npm cache (standard npm semantics).
- All injected files are local; no module ever phones home.
- The optional version-check (§9.4) has a 1-second timeout and silently no-ops on failure — `init`/`add`/`update` always succeed offline if the package is cached.

### 13.5 `.agents/` Already Exists / Partial Install

- Re-running `init` on a directory that already has a complete `.agents/` is allowed (idempotent).
- Re-running on a partial `.agents/` (e.g., user `rm`-ed half a module): `init` reports "partial install detected" and offers to fill missing files at recorded hashes; user-modified files are left alone.

### 13.6 Memory Log Rotation

Append-only `sessions/<date>.md` is a footgun for long-lived projects. v0.1 mitigation:

- The `memory` SKILL instructs the agent to summarize-and-truncate any `sessions/<date>.md` exceeding 500 lines into `topics/` files, then leave a one-line stub citing the rotation timestamp.
- No automatic rotation in v0.1; users who want hard caps add a cron / git hook themselves.

### 13.7 Network-Restricted CI

`harless doctor --offline` skips the version check and the npm registry probe; useful in air-gapped CI.
