## 1. Project Statement

`harless` is an **agent-agnostic, minimal AI Coding Harness kit**. Through `npx harless init` / `add`, it injects 8 capability modules into a user's project as plain markdown files and shell scripts. Any AI agent that respects the [`AGENTS.md`](https://agents.md) convention (Claude Code, Cursor, Codex CLI, Copilot CLI, Gemini CLI, ÔÇŽ) gains an immediate, complete production loop:

> Skills ┬Ě Spec ┬Ě Loop ┬Ě Memory ┬Ě Browser-Debug ┬Ě Orchestration ┬Ě Review ┬Ě Simplify

The goal is not to replace heavy alternatives (Superpowers, OpenSpec, claude-mem, agentation, etc.) but to provide the **smallest possible complete subset** that global Web developers can adopt at near-zero cost, and that is easy to learn, extend, and fork.

### 1.1 Non-goals (v0.1)

- Replacing Superpowers / OpenSpec / claude-mem feature-by-feature
- Self-built MCP server (deferred to v0.2+)
- Slash commands, hooks, plugins, subagent-specific configurations
- SaaS backend, telemetry, accounts, online dashboards
- Multi-OS matrix beyond macOS + Linux (Windows deferred to v0.2)
- A spec validator (deferred to v0.2+)

## 2. Locked Architectural Decisions

| # | Decision | Rationale |
|---|---|---|
| D1 | **Agent-agnostic** via `AGENTS.md` + Read tool + non-interactive CLI | Avoids per-agent adapter sprawl; serves global developers |
| D2 | **CLI-injected files (shadcn/OpenSpec style)** via `npx harless init` / `add` | Code lives in user's repo; trivial fork & customize |
| D3 | **Two primitives only**: Markdown + Shell/Node scripts | Keeps the surface tiny and learnable |
| D4 | External MCP only ÔÇö reuse `chrome-devtools-mcp`, no self-built MCP | YAGNI; community already maintains it |
| D5 | **Web-first defaults + generic degradation** | Auto-detect Next/Vite/React; degrade via `--stack=generic` |
| D6 | **Smoke-tested on Tier 1 = CC, Cursor, Codex CLI** | Three formats represent the field; others are AGENTS.md-compatible |
| D7 | **No daemons, no hooks, no global config writes** | Zero side effects on user's machine |
| D8 | Project name = **`harless`** | Short, unique, self-evidently "harness minus weight" |

### 2.1 Five Design Axioms (Inviolable)

1. **Zero runtime** ÔÇö no daemons, no port listeners, no background workers.
2. **Two primitives** ÔÇö only Markdown files and shell/Node scripts; external MCP only via reuse.
3. **Code belongs to the user** ÔÇö files written by `add` are user-owned; `update` uses a 3-way diff merge, never a clobber.
4. **AGENTS.md is the protocol** ÔÇö all cross-agent compatibility flows through one file at the project root.
5. **YAGNI ruthlessly** ÔÇö no slash commands, no self-MCP, no plugins, no GUI, no SaaS in v0.1.

## 4. CLI Surface

### 4.1 Commands

| Command | Purpose | Idempotent |
|---|---|---|
| `harless init` | Detect stack, interactively select modules, write files | Ôťů |
| `harless add <module>` | Add one module after init | Ôťů |
| `harless remove <module>` | Remove one module (confirm) | Ôťů |
| `harless update` | 3-way hash merge with new templates | Ôťů |
| `harless doctor` | Inspect environment, agent config, MCP, permissions | Ôťů (read-only) |

**Not in v0.1:** `login`, `publish`, `share`, `ai`, `run`, `watch`, `spec new`, `memory query`. The CLI is **not** a second agent UI ÔÇö agents call scripts and read files directly.

### 4.2 `init` Behavior

1. Detect stack (`package.json` heuristics: `next`, `vite`, `react`, `remix`, `astro`, `svelte`).
2. Detect package manager (`pnpm-lock.yaml` / `yarn.lock` / `package-lock.json`).
3. Interactive multi-select: 6 modules pre-checked (`skills, spec, loop, memory, browser-debug, review`), 2 unchecked (`orchestrate, simplify`).
4. Interactive multi-select: smoke-tested agent (CC, Cursor, Codex CLI, Other).
5. Write files; merge `AGENTS.md` and `.mcp.json` if present.
6. Print "Next steps" with verification command.

**Non-interactive form:** `--yes --modules=... --agents=...` for CI.

### 4.3 `update` ÔÇö Three-Way Hash Merge

Each templated file's SHA-256 is recorded in `.harness/config.json` at install time. On `update`:

| Disk hash vs old template | New template vs old template | Action |
|---|---|---|
| same | changed | silent overwrite |
| changed | same | leave disk untouched |
| changed | changed | print 3-way diff; user picks `[k]eep / [o]verwrite / [m]erge` |
| missing | n/a | offer to recreate |

**No automatic clobber on conflict, ever.**

### 4.4 `doctor` Checks

- Node Ôëą 18
- `AGENTS.md` exists at root and contains the `harless` managed block
- `.harness/scripts/*.sh` are executable
- If `browser-debug` enabled: `.mcp.json` lists `chrome-devtools-mcp`; Chrome reachable
- For each declared agent: MCP read-path covered (or printed snippet for the user to paste, e.g. Codex CLI's `~/.codex/config.toml`)

Each check returns Ôťô / ÔÜá / ÔťŚ with a single-line remediation.

## 6. Eight Modules

Each module is **1 SKILL.md + (optional) 1 script**, ÔëĄ 120 LOC markdown and ÔëĄ 80 LOC script.

### 6.1 `skills` ÔÇö behavioral skills (Superpowers minimal subset)

5 sub-SKILLs only, chosen for "agent wouldn't think of it + biggest quality impact":
- `brainstorming` (intent before code)
- `tdd` (test-first discipline)
- `systematic-debugging` (no random guesses)
- `verification-before-completion` (evidence over assertions)
- `receiving-review` (structured response to review feedback)

### 6.2 `spec` ÔÇö Spec-Driven Development (OpenSpec minimal)

Three-step workflow: idea Ôćĺ `spec.md` (what) Ôćĺ `plan.md` (how) Ôćĺ `tasks.md` (atomic tasks). Templates only; no validator in v0.1. Agent copies `.template/` to a dated change folder.

### 6.3 `loop` ÔÇö Ralph self-correcting loop

`scripts/loop.sh` (< 40 LOC bash):

```bash
#!/usr/bin/env bash
set -euo pipefail
WORK="${1:?need work-dir}"; cd "$WORK"
MAX="${MAX_ITER:-20}"
AGENT_CMD="${AGENT_CMD:-claude -p}"
for i in $(seq 1 "$MAX"); do
  echo "=== iter $i ==="
  $AGENT_CMD "$(cat goal.md)" | tee "iter-$i.log"
  if bash check.sh; then echo "Ôťô done @ iter $i"; exit 0; fi
done
echo "ÔťŚ max iterations reached"; exit 1
```

`AGENT_CMD` env var makes the script agent-agnostic.

### 6.4 `memory` ÔÇö cross-session memory (claude-mem minimal, zero daemon)

- Write: `scripts/compact.sh` calls `$AGENT_CMD` to summarize the current conversation into markdown ÔÇö synchronous, no background worker.
- Read: SKILL.md instructs the agent to `ls`/`grep` `.harness/memory/topics/` at session start.
- Filename = retrieval key (`auth.md`, `deploy-netlify.md`, ÔÇŽ).
- Secret protection: SKILL.md's redaction checklist; README recommends gitignoring `memory/sessions/`.

### 6.5 `browser-debug` ÔÇö Web debugging via `chrome-devtools-mcp`

- Zero self-built code; we just merge `chrome-devtools-mcp` into `.mcp.json`.
- The skill's value is **discipline**: snapshot before action, scroll-into-view before click, abandon after 4 failed attempts. (See cursor-ide-browser conventions.)

### 6.6 `orchestrate` ÔÇö parallel subagent dispatch (agentation minimal)

- `scripts/dispatch.sh` reads `tasks.yml` (id, prompt, cwd) and runs them in parallel via `xargs -P` or background jobs, each calling `$AGENT_CMD`.
- v0.1: parallel-only, no DAG. DAGs decompose into multiple sequential dispatches.
- Default-off in `init`.

### 6.7 `review` ÔÇö independent diff review

`scripts/review.sh` (< 60 LOC bash) spawns a fresh `$AGENT_CMD` process with **only the checklist + diff** (no main-session context bias):

```bash
REF="${1:-HEAD}"
DIFF=$(git diff "$REF")
CHECKLIST=$(cat .harness/skills/code-review/SKILL.md)
echo "$CHECKLIST

=== DIFF ===
$DIFF" | $AGENT_CMD > .harness/review-$(date +%s).md
```

Independent process (not subagent) keeps it cross-agent.

### 6.8 `simplify` ÔÇö code simplification scan

Pure-skill primarily; `scripts/simplify.sh` is an optional accelerator following the same independent-agent pattern as `review`. Default-off in `init`.

### 6.9 Common Script Conventions

All scripts honor:
1. `$AGENT_CMD` env (default `claude -p`) for cross-agent.
2. `set -euo pipefail` for fail-loud behavior.
3. stdout artifacts to `.harness/<module>/<timestamp>.{log,md}`.
4. exit 0 = success, Ôëą 1 = needs human.
5. No deps beyond `git`, `bash`, `node`. No `jq`/`yq`/`parallel` ÔÇö small inline alternatives.

## 7. Cross-Agent Compatibility

### 7.1 Capability Matrix

| Agent | Reads AGENTS.md | Read tool | Terminal tool | Non-interactive CLI | MCP path |
|---|---|---|---|---|---|
| Claude Code | Ôťů | Ôťů | Ôťů | `claude -p` | `.mcp.json` |
| Cursor | Ôťů | Ôťů | Ôťů | `cursor-agent -p` | `.cursor/mcp.json` or `.mcp.json` |
| Codex CLI | Ôťů | Ôťů | Ôťů | `codex exec` | `~/.codex/config.toml` (global) |
| Copilot CLI | Ôťů | Ôťů | Ôťů | `copilot -p` | varies |
| Gemini CLI | Ôťů | Ôťů | Ôťů | `gemini -p` | `.gemini/settings.json` |

### 7.2 Four Differences We Resolve

**Difference 1 ÔÇö MCP config paths.** Write `.mcp.json` (covers CC + Cursor). For Codex/Copilot/Gemini, print a paste-snippet in `.harness/README.md`. **Never modify global config files.**

**Difference 2 ÔÇö `$AGENT_CMD` syntax.** Stored in `config.json:defaultAgentCmd`; scripts honor env override. Default = `claude -p`. Each script's header lists alternative invocations.

**Difference 3 ÔÇö Skill discovery vs manual reference.** SKILL.md frontmatter benefits CC/Cursor's native discovery. AGENTS.md's "Skills Index" table benefits agents without native skill systems. Same SKILL.md serves both.

**Difference 4 ÔÇö AGENTS.md coexistence with other tools.** Marker-delimited managed block (`<!-- BEGIN/END harless v0.1 -->`). `update` only touches that block. If openspec/superpowers blocks are detected, `doctor` notes coexistence and links to a reconciliation doc.

### 7.3 Verification Gate (release red line)

Run on each Tier 1 agent: ask `"Please help me turn this idea into a design spec: a todo list app."` Expected: agent reads `.harness/skills/brainstorming/SKILL.md` **before** acting, then follows it. **A failure on any Tier 1 agent blocks release.**

### 7.4 Compatibility Tiers

- **Tier 1 (smoke-tested):** Claude Code, Cursor, Codex CLI
- **Tier 2 (AGENTS.md-compatible, community-verified):** Copilot CLI, Gemini CLI
- **Tier 3 (best-effort):** any agent reading AGENTS.md with shell + Read tools

### 8.3 Session Phases

| Phase | Trigger | Required action | Artifacts |
|---|---|---|---|
| Start | first user message | read AGENTS.md; if `memory` enabled, `ls memory/topics/` and grep | `memory/topics/` (read) |
| Mid | each task | route via Skills Index; call scripts via `$AGENT_CMD` | `specs/`, `loop/`, `review-*.md` |
| End | user says "done" / completes a TodoWrite root task | `compact.sh` | `memory/sessions/<date>.md` (append), `memory/topics/*.md` (upsert) |

Session end is **not auto-detected**. SKILL.md tells the agent which signal words and structural completions trigger compaction. This deliberately avoids hooks (which would break agent-agnosticism).

### 8.4 Failure Modes (decision matrix)

| Scenario | v0.1 behavior | Not done (over-engineering) |
|---|---|---|
| `loop.sh` exhausts MAX_ITER | exit 1, prints last log path | no auto prompt-rewrite, no agent swap |
| `review.sh` finds severe issue | writes `review-<ts>.md`; main agent decides | no commit blocking, no git hook |
| `compact.sh` may include secrets | redaction checklist + README gitignore tip | no automatic secret scanner (v0.2+) |
| Co-installed with OpenSpec | `.specs/` and `.openspec/` coexist independently | no migration tool |
| `chrome-devtools-mcp` version drift | `doctor` prints versions and upgrade hint | no version pinning, no vendoring |
| User edited a SKILL file | `update` 3-way diff; user resolves on conflict | no "protected mode", no intent tracking |
| Tier 1 agent inconsistency | release blocked; Tier 2/3 Ôćĺ issue | no runtime capability probing |

Stance: **observable + user-decided** beats **automatic fallback**.

## 9. Error Handling, Testing, Distribution

### 9.1 Error-Handling Principles

1. **Fail loud, fail early** ÔÇö `process.exitCode` in CLI; `set -euo pipefail` in scripts; preconditions in SKILLs.
2. **Never silently mutate** ÔÇö show a path before any write; require explicit choice on conflict; `--yes` exits 2 on conflict.
3. **Recoverable by hand** ÔÇö delete `.harness/` and re-`init` always recovers; everything is plain text in git.

### 9.2 Test Pyramid

**Unit (vitest):** `mergeAgentsMd`, `mergeMcpJson`, `threeWayHashMerge`, `detectStack`. Coverage Ôëą 80% line, Ôëą 70% branch. AGENTS.md rendering uses snapshot tests.

**Integration (`execa` + temp dirs):** 8 mandatory scenarios:
1. Empty dir `init` (default selection)
2. `init` over existing AGENTS.md containing an `openspec` block
3. `init Ôćĺ add orchestrate Ôćĺ remove orchestrate`
4. `init` Ôćĺ user edits one SKILL Ôćĺ `update` (template unchanged) Ôćĺ user edit preserved
5. `init` Ôćĺ user edits one SKILL Ôćĺ `update` (template also changed) Ôćĺ conflict path; `--yes` exits 2
6. Re-`init` (idempotency)
7. `doctor` exit codes across 7 scenarios (all green / missing perms / missing MCP / missing AGENTS block / etc.)
8. `.mcp.json` with other servers + add `chrome-devtools-mcp` (merge correctness)

**E2E (manual, recorded):** per Tier 1 agent, a checklist + screen recording in `docs/smoke/<agent>.md`. All three must be green to release.

**Out of scope:** real agent CI, semantic LLM-output assertions, multi-OS matrix.

### 9.4 Versioning, Release, Upgrade

- **SemVer.** v0.x may break template structure; v1.0+ guarantees Ôëą 2-minor migration paths.
- **Release:** tag `v0.x.y` Ôćĺ CI runs unit + integration Ôćĺ publish npm. E2E recordings must be merged into `docs/smoke/` first (manual gate).
- **Upgrade nudge:** silent npm-registry version check on each invocation (< 1 s timeout, fails silent). One-line notice if behind. **No telemetry.**
- **Deprecation:** any template-file removal must warn through `update` for at least one minor before removal.

### 9.5 Documentation

VitePress site + README:
- README ÔÇö 10-minute onboarding (install Ôćĺ init Ôćĺ first brainstorm Ôćĺ first loop)
- Getting Started ÔÇö same, longer, with troubleshooting
- One page per module
- "Compared with other Harnesses" ÔÇö honest positioning vs Superpowers / OpenSpec / claude-mem / agentation
- "Extending harless" ÔÇö how to remove / add / fork modules

**Out of scope:** interactive playground, online config builder, blog, SaaS console.

### 9.6 Privacy & Compliance

- **Zero telemetry.** No data leaves the user's machine. The only network call is the npm-registry version check (npm itself is already online).
- README states this prominently: "no telemetry, no account, no network calls except npm registry".

### 9.7 License & Governance

- **MIT** ÔÇö most permissive, aligned with "fork freely".
- **Governance** ÔÇö single maintainer in v0.1; CONTRIBUTING covers issue/PR rules; no SLA during v0.x.
- **Code of Conduct** ÔÇö Contributor Covenant (one file).

### 9.8 Roadmap (v0.2+ candidates, not committed)

| Candidate | Rationale to defer | Trigger to promote |

## 10. Definition of Done (v0.1)

A v0.1 release is "done" when **all** of the following are true:

1. CLI source ÔëĄ 500 LOC; injected templates total ÔëĄ 1800 LOC.

3. All 8 modules' SKILL.md present; 5 modules' scripts present and executable.
4. Unit tests Ôëą 80% line / Ôëą 70% branch coverage on the four merge/detect functions.
5. All 8 integration scenarios green in CI.
6. Tier 1 smoke recordings (CC + Cursor + Codex CLI) merged into `docs/smoke/`.
7. README + Getting Started + 8 module pages + "Compared withÔÇŽ" page + "ExtendingÔÇŽ" page published.
